# v0.1.1 Fixes & Polish — Design

**Date:** 2026-07-02
**Status:** Approved design pending user review
**Follows:** v0.1.0 public release. Precedes a separate "SNESOffset" architecture effort (out of scope here).

## Goal

Fix the two field bugs found during first real-world testing, surface emulator
attach status so failures are legible, apply small default/UX improvements, and
rename the DLL — shipping as **v0.1.1**.

## Context (evidence from testing)

- **Bug A — spurious exit & moon counts on game start.** Loading a save with 24
  exits produced a spurious `+1` on both the exit and moon counters (on
  different frames).
- **Bug B — laptop counted nothing.** RetroArch x64 1.22.2 + snes9x_libretro
  core **1.63 `185488c`**. Attach failed silently: no offset entry for that core
  build. Dev machine worked (attached a different way — standalone snes9x).
- Root causes and reference comparison against **kaizosplits** (the project these
  counters were ported from) are recorded below.

## Fixes

### 1. Exit counter — adopt kaizosplits event detection (Bug A, exit)

**Root cause:** [`ExitCounter`](../../../src/LiveSplit.SmwCounters/Counters/ExitCounter.cs)
watches the `$1F2E` ExitsCompleted **save byte** and counts on any increase. A
save load populates `$1F2E` `0 → 24`, read as a rising edge → spurious `+1`.
kaizosplits never reads `$1F2E`; it detects the exit **event** via `exitMode`.

**Fix:** rewrite `ExitCounter` to match kaizosplits
([Watchers.cs](https://github.com/amcknight/kaizosplits/blob/main/Components/SMW/SMW/Watchers.cs)):

- Read `exitMode` at **`$0DD5`** (was `$1F2E`).
- Count when `exitMode` **shifts** to a valid exit value:
  `HasPrevious && curr != prev && curr != 0 && curr != 128 → Value++`.
- Keep the existing `PreviousByte` baseline-clear-on-detach behavior (immune to
  load: first post-attach sample only sets the baseline).

This is event-based (save-load immune), matches the proven reference, and
sidesteps the `$1FE2` "+4 per exit" pitfall the old comment described.

### 2. Moon counter — in-level game-mode gate (Bug A, moon)

**Root cause:** [`MoonCounter`](../../../src/LiveSplit.SmwCounters/Counters/MoonCounter.cs)
watches `$13C5` (per-scene moons) for increases with **no game-mode gate**;
load-time / overworld reads of that byte fire spuriously. kaizosplits has **no
moon counter**, so there is no reference to copy — this stays our design.

**Fix:** gate moon counting to actually being in a level.

- Read `levelStart` at **`$1935`**; "in level" = `levelStart == 1`
  (kaizosplits' `InLevel` definition).
- When **not** in a level, clear the moon baseline (`previousMoon.Clear()`) and
  return without counting — so the save-load populate and overworld reads never
  register an edge.
- When in a level, keep the existing `$13C5`-increase detection and the
  All / PerLevel / PerRoom dedupe logic unchanged.

Death is unaffected — it already matches kaizosplits (`playerAnimation == 0x09`).

### 3. Offset coverage — add the missing snes9x_libretro core build (Bug B)

**Root cause:** the offset tables are a frozen snapshot of kaizosplits and have
drifted. kaizosplits already carries the laptop's core.

**Fix:** add to
[`Offsets.CoreMem`](../../../src/LiveSplit.SmwCounters/Snes/Offsets.cs), verbatim
from kaizosplits' `Offset.cs`:

```csharp
{ "snes9x_libretro.dll 1.63 185488c", 0x3BB164 }, // same 1.63 offset as 49f4845
```

Also do a targeted diff of our core/emulator tables against kaizosplits' current
`Offset.cs` and add any other clearly-missing, reference-verified entries
(data-only; no guessed values).

### 4. Emulator status readout (diagnostics + UX)

There is currently no way to see whether the component attached — `SnesEmu`
computes `Describe()` and `LastError` but nothing surfaces them. This is why
Bug B failed silently.

**Fix:** add a read-only **status line to the settings dialog** showing the
current attach state (e.g. `Attached: snes9x 1.60`, `no emulator found`, or
`no WRAM offset for core 'snes9x_libretro.dll 1.63 185488c'`).

- Add a `Label` to `SmwCountersComponentSettings` and a `SetStatus(string)`
  method.
- The component already polls every 15 ms; each poll, push
  `emu.Describe()` (or the gated-off reason) to the settings label.
- Must not throw when settings aren't open.

### 5. Default changes

- `ResetOnSplitsReset` default **false → true**
  ([SmwCountersComponentSettings.cs:25](../../../src/LiveSplit.SmwCounters/UI/Components/SmwCountersComponentSettings.cs#L25)).
- Default `RowHeight` **40 → 50** — update **both** the initializer
  ([:23](../../../src/LiveSplit.SmwCounters/UI/Components/SmwCountersComponentSettings.cs#L23))
  and the settings-parse fallback
  ([:321](../../../src/LiveSplit.SmwCounters/UI/Components/SmwCountersComponentSettings.cs#L321)).

### 6. Settings & display polish

- **Counter order:** reorder the registry in `SmwCountersComponent` ctor from
  `Deaths, Moons, Exits, Jumps` to **`Deaths, Exits, Moons, Jumps`** so Exits
  sits above Moons (affects both the settings list and the drawn row order).
- **Moon dedupe radios:** in `BuildUi`, place a counter's extras panel tight
  under its own row (reduce the vertical gap) so the moon radio buttons read as
  attached to the Moons row. Final pixel spacing to be eyeballed in-app.
- **Tooltip:** shorten `SmwCountersComponentFactory.Description` from
  "Watches SNES WRAM in an emulator process and counts SMW deaths, moons, and
  more — pick which to show." to something like
  **"Counts SMW deaths, exits, jumps, and 3-up moons from your emulator."**

### 7. Packaging & docs

- **Rename the DLL** to `SMWCounters.dll` via `<AssemblyName>SMWCounters</AssemblyName>`
  in the csproj. LiveSplit's `ComponentManager` loads all `*.dll` with no prefix
  filter (verified), so this is safe. Update:
  - the release workflow's DLL search/stage step (currently filters
    `LiveSplit.SmwCounters.dll`), and
  - the README's DLL filename references.
- **README menu path fix:** category is `ComponentCategory.Other`, so the menu
  path is **Edit Layout → + → Other → SMW Counters** (README currently says
  "Information" — wrong).
- **Release notes:** add a `body:` to the `action-gh-release` step with the
  install steps (copy `SMWCounters.dll` into `LiveSplit/Components/`, then
  Edit Layout → Other → SMW Counters) so the instructions live on the Release
  page.

### 8. Version

Bump assembly `<Version>` **0.1.0 → 0.1.1**; ship on tag `v0.1.1` (tagging is the
user's manual step, as before).

## Non-goals (explicitly deferred)

- **SNESOffset shared-architecture project** (centralized offset data and/or
  runtime WRAM discovery via RetroArch `READ_CORE_MEMORY` + signature scan).
  Its own brainstorming → spec → build effort after v0.1.1.
- Reverse-engineering offsets for arbitrary unlisted core builds.
- Code-signing the DLL (SmartScreen "unverified" warning is expected for
  unsigned components; out of scope).

## Testing approach

- Exit and moon fixes are unit-testable with a fake `ISnesMemory` feeding scripted
  byte sequences (attach → file-select populate → in-level events), asserting no
  spurious counts on load and correct counts on real events. TDD these.
- Offset add, defaults, rename, status label, and docs are verified by build +
  in-app check.

## Success criteria

1. Loading a save with N exits produces **no** spurious exit or moon count.
2. A real level exit increments Exits once; a collected 3-up moon increments
   Moons once (in-level only).
3. The settings dialog shows a live emulator-attach status line.
4. The laptop's snes9x_libretro 1.63 `185488c` core attaches and counts.
5. Built DLL is `SMWCounters.dll`, version `0.1.1`; README/menu path correct;
   Release notes carry install instructions.
