# v0.1.3 — Laptop-review fixes (Settings cleanup + banking toggle + gold alert + exit timing) — Design

**Date:** 2026-07-03
**Status:** Approved design pending user review
**Follows:** 0.1.2 (banked counters + Powerups + jump gate, shipped as the
pre-stable test release). 0.2.0 stays reserved for first stable. This ships as
**0.1.3**.

## Goal

Address the batch of findings from the on-laptop review of 0.1.2:

- The Settings dialog is cluttered and inconsistent (per-counter label textbox
  nobody uses, mixed radio/combobox controls, verbose button text).
- The banked-counter alert reads as red ("bad") when it should read as gold
  ("carrying something good, not yet locked in").
- Banking-on-death should be optional per counter, not always-on.
- The Exit counter's alert fires at *exit-write* instead of at the *goal*, then
  self-clears — because it collects on the late `exitMode` signal rather than
  the early finish event.

Two larger ideas surfaced in review are **explicitly deferred** (see Non-goals):
weighted powerup counting (Cape/Fire = 2) and checkpoint counting.

---

## 1. Settings UI cleanup

All changes are in `SmwCountersComponentSettings` (and the row-wiring in
`SmwCountersComponent`). No behavioral change to counting from this section.

### 1a. Remove the per-counter label textbox and all override machinery

The label textbox let a user replace a counter's icon with typed text. It is
barely used and is the main source of clutter. Remove it entirely:

- Delete `row.Label` (the per-row `TextBox`) and its wiring.
- Delete the `labels` dictionary, `GetLabelOverride`, `SetLabelOverride`, and the
  `CounterLabels` read/write in `SetSettings`/`CreateSettingsNode`.
- Delete `OverrideTextOrNull` in the component; `Update`/`DrawGeneral` always
  draw the counter's `DefaultIcon` (or `DefaultLabel` text when a counter has no
  icon). The icon-vs-override branch in `DrawGeneral` simplifies to icon-or-label.
- **Back-compat:** old layouts may still contain a `<CounterLabels>` node.
  `SetSettings` simply stops reading it (ignored, no crash). Any prior override
  text is lost — accepted; the feature is being removed.

### 1b. Replace it with a small editable value box

In the freed horizontal space, each row gets a **small numeric TextBox
(~45px wide)** adjacent to the Reset button that shows the counter's current
value and lets the user see/seed it (e.g. continuing a run from another session):

- On panel build/refresh, the box text is set from the counter's current
  `Value`.
- On commit (the box's `Leave` event, and `Enter` key), parse the text as an int;
  if valid, call the counter's new `SetValue(int)`. Invalid/empty input reverts
  the box to the current value.
- Not a live ticker — it does not fight the user's typing. Editing happens while
  the timer is stopped (value is static, usually 0), which is the intended use.

This requires a new interface method (see §6).

### 1c. "Reset value" button → "Reset"

Rename the button text. Reposition next to the enable checkbox now that the
textbox slot is gone.

### 1d. Alignment combobox → radios

Replace the `cboAlignment` `ComboBox` with three `RadioButton`s (Left / Center /
Right) so all multi-choice settings use the same control style as Moon's dedupe
mode. Selection maps to the existing `HAlignment` enum exactly as today.

### 1e. Moon dedupe radios reflow

With the textbox gone, the Moon dedupe radio panel (All / Per level / Per room)
moves onto the counter's own row, to the **right of the Reset button / value
box**, instead of on a separate line below. (Implementation: it remains the
counter's `Extras` control; only its layout position changes.)

### 1f. Grey out a disabled counter's row controls

When a counter's enable checkbox is unchecked, disable (grey out) that row's
Reset button, value box, and `Extras` control. Wire this in the
`Enable.CheckedChanged` handler and apply the initial state on build.

### 1g. Row-height default 50 → 45

Change the default `RowHeight` from 50 to 45 in both the field initializer and
the `SetSettings` parse default. Existing layouts store an explicit `RowHeight`
and keep their value; only new layouts get 45.

## 2. "Bank on save" toggle (per banked counter)

Today banked counters (Exits, Powerups) always do die-to-discard and always show
the alert while carrying an unbanked collect. Make this **optional per counter**.

### Behavior

- New per-counter boolean, **default ON**, shown **only on banked counters**
  (Exits, Powerups) as a `CheckBox` labelled **"Bank on save"** occupying that
  row's `Extras` slot.
- **ON** = today's behavior: `total` shows in gold while `total != saved`;
  death-before-save reverts (`total = saved`); a bank locks it in.
- **OFF** = plain permanent tally: every collect sticks immediately, no
  death-revert, never alerts.

### Implementation

- `BankedCounter` gains `public bool Banked { get; set; } = true;`.
- `BankedCounter.Poll` becomes:
  ```
  if (DetectDeath(memory))   { total = saved; }         // no-op when !Banked (total==saved)
  if (DetectCollect(memory)) { total++; if (!Banked) { saved = total; } }
  if (DetectBank(memory))    { saved = total; }          // harmless when !Banked
  ```
  With `Banked == false`, `total == saved` always holds, so `ValueIsAlert` is
  always false and death does nothing — exactly the plain-tally semantics.
- The component sets `bankedCounter.Banked = Settings.IsBankOnSave(id)` at the
  top of each `Poll()` for banked counters (cheap; keeps the counter in sync with
  the setting without extra events).
- `SmwCountersComponent.BuildExtras`: for a `BankedCounter`, return a
  "Bank on save" `CheckBox` (`Checked = Settings.IsBankOnSave(id)`, default true;
  `CheckedChanged → Settings.SetBankOnSave(id, checked)`). Moon still returns its
  radio panel.

### Serialization

- Store the OFF set only (default-ON, so absence = ON): `HashSet<string>
  bankDisabled` in Settings.
- Serialize as `<BankDisabled><Counter>id</Counter>…</BankDisabled>`; read it
  back in `SetSettings` (absent → empty set → all banked-on).
- `IsBankOnSave(id) => !bankDisabled.Contains(id)`;
  `SetBankOnSave(id,true) => bankDisabled.Remove(id)`, `false => Add(id)`.

## 3. Alert color → gold

The alert color currently uses `state.LayoutSettings.BehindLosingTimeColor`
(theme "losing" red), which reads as *bad*. A carried-but-unbanked collect is
*good* — you got something, it just isn't locked in yet.

- In `SmwCountersComponent.DrawGeneral`, change the value color to:
  `c.ValueIsAlert ? state.LayoutSettings.BestSegmentColor : textColor`
  (theme gold in the default LiveSplit theme, still user-overridable). One line.
- Remains a static color state (not an animation), as before.

## 4. Exit timing fix (collect on the true finish, not on exit-write)

### The bug

The Exit counter collects on `exitMode` (`$0DD5`) shifting to a valid value —
kaizosplits' late `LevelExit` signal, which fires when the game *writes the
exit*, after the fanfare. Result: the gold alert appears at exit-write, not at
the goal, and clears a beat later when `$1F2E` banks. The user observed exactly
this ("turned gold on exit, then reverted on its own").

### The fix

Collect on the **early finish event** (kaizosplits' finish detection), keep
banking on `$1F2E`. The gold window then spans goal → save correctly, and an
after-goal death still reverts.

**Finish types counted:** `Goal || Key || Orb || Boss`. **Switch Palace is
excluded** — palaces (in vanilla) don't increment `$1F2E`, so collecting on one
would leave the alert stuck gold forever with no bank to clear it. (In hacks a
palace may write an exit, but a post-palace death effectively never happens, so
the risk of the stuck-alert outweighs the rare missed count.)

**New WRAM reads** (addresses from kaizosplits `Memory.cs`):

| Name        | Addr    | Use                                             |
|-------------|---------|-------------------------------------------------|
| `fanfare`   | `$0906` | level-clear fanfare trigger                     |
| `io`        | `$1DFB` | SPC700 I/O port; 3=Orb, 4=Goal, 7=Key           |
| `bossDefeat`| `$13C6` | 0 = boss not (yet) defeated                     |

**Detection** (kaizosplits `Compare` semantics: `StepTo(p,c,to)` ⇔
`c==to && p+1==c`; `ShiftTo(p,c,to)` ⇔ `p!=to && c==to`):

```
fanfareStep = StepTo(fanfare, 1)          // fanfare 0 -> 1
bossUndead  = (bossDefeat == 0)
Goal        = fanfareStep && bossUndead && io != 3
Orb         = ShiftTo(io, 3) && bossUndead
Key         = ShiftTo(io, 7)
Boss        = fanfareStep && !bossUndead
collect     = Goal || Orb || Key || Boss
```

`DetectCollect` reads `fanfare` and `io` via `PreviousByte` watchers and
`bossDefeat` as a current value; clears those watchers on a failed read and in
`ClearDetectors`. `DetectBank` (on `$1F2E`) is unchanged. The old
`exitMode`/`$0DD5` collect is removed.

**Known limitation (unchanged from before):** re-completing an already-saved
exit fires the collect but does not increment `$1F2E` (the game only counts new
exits), so it would leave the alert gold until a death or reset. This does not
occur in normal fresh-run play; documented, not fixed.

## 5. Powerups — confirm the transform-based scheme, live-verify swaps

No detection change is planned; this section pins the **agreed semantics** and a
verification step.

### Agreed rule

A powerup counts when it **changes Mario's form** — the existing collect signal
(`playerAnimation $0071` rising into `{2,3,4}` = mushroom / feather / flower,
gated to in-level). This already implements the "banked-but-unused doesn't count
until used, and can't be gamed" ideal: a stockpiled (reserve-box) powerup counts
only if/when it later transforms Mario, and each count is welded to one physical
transformation, so it can't be double-fired.

Case table (target behavior):

| Situation                                   | Counts |
|---------------------------------------------|:------:|
| Small → mushroom → Big                       |  +1    |
| Big → mushroom → reserve box                 |  +0    |
| Deploy reserve mushroom, grab while Small    |  +1    |
| Fire → feather → Cape (swap)                 |  +1 \* |
| Cape → flower → Fire (swap)                   |  +1 \* |
| Take damage → downgrade                       |  +0    |
| Fire → flower (already Fire) → reserve        |  +0    |

### Live verification (must do during implementation)

\* The cape↔fire **swap** is the user's must-have. It is the one case not
confirmed empirically: verify in an emulator that grabbing a feather while Fire
(and a flower while Cape) fires the `$0071` → 3 / → 4 animation.

- **If it fires:** no code change; the current `PowerupCounter` already counts
  it. Just document.
- **If it does NOT fire** (silent `$0019` 2↔3 swap): add a supplementary collect
  signal — a lateral change of `$0019` between 2 and 3 while in-level — deduped
  against the animation signal within the same poll so a single swap counts once.

Powerups otherwise gain nothing new here except being subject to the Bank-on-save
toggle (§2) and the gold color (§3), like Exits.

## 6. Interface / wiring changes

- `ISmwCounter` gains `void SetValue(int value);`.
  - Simple counters (`DeathCounter`, `MoonCounter`, `JumpCounter`): set the
    backing count field (expose the setter).
  - `BankedCounter`: `SetValue(v) { total = v; saved = v; }` (clears any alert).
- The component's per-counter row tuple grows to carry value access:
  `(string Id, string DefaultLabel, Control Extras, Action ResetValue,
    Func<int> GetValue, Action<int> SetValue)`, wired as
  `GetValue = () => c.Value`, `SetValue = v => c.SetValue(v)`.
- `SmwCountersComponentSettings.BuildUi` consumes the new tuple to populate and
  commit the value box, and adds the grey-out and reflow layout.

## 8. Testing (TDD, fake `ISnesMemory`)

**ExitCounter (new finish detection):**
- Goal: `fanfare` 0→1 with `bossDefeat==0`, `io!=3` → collect (total++, alert on).
- Orb: `io` →3 with `bossDefeat==0` → collect.
- Key: `io` →7 → collect.
- Boss: `fanfare` 0→1 with `bossDefeat!=0` → collect.
- Palace-equivalent / no-finish inputs → no collect.
- Bank on `$1F2E` increment (saved=total, alert off).
- After-goal death (finish, then `$0071`→9 before `$1F2E`) → total reverts, no net.
- Save-load `$1F2E` 0→24 with no pending finish → no collect, no phantom bank.
- `exitMode` shifts no longer trigger a collect.

**BankedCounter Bank-on-save toggle:**
- `Banked==true`: collect → alert on; bank → alert off; die before bank → revert.
- `Banked==false`: collect → total & saved advance together; `ValueIsAlert`
  always false; death does not revert; explicit bank is a no-op.

**PowerupCounter:**
- Existing in-level/off-level collect and bank (midway / `$1F2E`) tests still
  pass under the toggle default (ON).
- (After live check) swap case behaves per §5.

**SetValue:**
- Simple counter `SetValue(n)` sets `Value`.
- `BankedCounter.SetValue(n)` sets both tallies; `ValueIsAlert` false afterward.

**Settings serialization:**
- `BankDisabled` round-trips; absent node → all banked-on.
- Old layout with `<CounterLabels>` loads without error (node ignored).
- Old layout `RowHeight` preserved; new layout defaults to 45.

## 9. Non-goals / deferred

- **Weighted powerup counting** (Cape/Fire = 2 vs. mushroom = 1). Floated on the
  idea that Fire "contains" two powerups, but the damage-revert behavior it
  leans on is unreliable — a hit while Cape/Fire appears to drop straight to
  Small (and may be hack-dependent), not to Big — so the fixed-weight rationale
  is shaky. Deferred to 0.1.4+. Recorded wrinkle if revisited: a cape↔fire
  **swap** is lateral, so it must be +0, with +2 only when rising from
  Big-or-below — needs live-case work.
- **Checkpoint / midway counting.** Dropped, not merely deferred: CPs can't be
  lost on death, are often unavoidable in hacks (door-triggered), and the
  get-midway-then-rebeat low% loop is not worth detecting. "No skippable
  midways" stays a social/implied low% rule. (If ever revisited: count only a
  midway that raised `$0019`, i.e. actually made Mario big.)
- Reserve-box-exact powerup counting beyond the transform scheme (§5) — not
  needed; the transform scheme is the intended, clean definition.
- Live-updating value box (it reflects value at panel open, not a ticker).

## 10. Success criteria

1. Settings dialog: no label textboxes; a small value box per row that shows and
   can seed the count; "Reset" button; Alignment as radios; Moon radios beside
   Reset; disabled counters' controls greyed out; new layouts default to 45px.
2. "Bank on save" checkbox on Exits and Powerups (default ON). OFF makes the
   counter a plain permanent tally (no gold, no death-revert).
3. Banked alert draws in theme gold (`BestSegmentColor`), not red.
4. Exit alert appears **at the goal/orb/key/boss finish** and clears at save;
   after-goal death reverts; switch palaces never trigger it; save-load produces
   no phantom exit.
5. Powerup counting unchanged in behavior, with the cape↔fire swap verified live
   (fallback added only if the animation does not fire).
6. Version 0.1.3; all unit tests pass.
