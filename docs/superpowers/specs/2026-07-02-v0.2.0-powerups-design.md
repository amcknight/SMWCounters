# v0.2.0 — Banked counters (Powerups + Exits) + jump gate — Design

**Date:** 2026-07-02
**Status:** Approved design pending user review
**Follows:** v0.1.1 (tagged, all three fixes validated live). Ships as **v0.2.0**.

## Goal

Add a **Powerups** counter for low% categories (collect-but-don't-use: you must
die to discard a powerup before "banking" a save), and apply the same
collect-then-bank model to the **Exit** counter so after-goal deaths don't
inflate it. Extract the shared logic into a `BankedCounter` base. Also fix the
jump counter's spurious title/overworld counts.

## The banked-counter model

Two internal tallies per banked counter:
- **`total`** — the displayed value; bumped on a **collect** event.
- **`saved`** — the "banked" count.

Shared rules:
- **Collect** → `total++`.
- **Die** (`playerAnimation $0071` rising-edge to `9`) → `total = saved`
  (discard everything collected since the last bank — "die-to-discard").
- **Bank** → `saved = total` (lock in what you're carrying).

`Value => total`. `ValueIsAlert => total != saved`.

During normal play `total == saved` (a collect is banked moments later), so the
counter reads like a plain tally. The distinction only matters in the window
between collecting and banking — which is exactly where a death should void the
collect.

## The two banked counters

**PowerupCounter** (new, `Id = "powerups"`, off by default, mushroom icon):
- **Collect:** `playerAnimation $0071` rising-edge into `{2,3,4}`
  (mushroom / feather / flower), **gated to in-level** (`levelStart $1935 == 1`)
  so overworld/load/garbage animation values can't false-count.
- **Bank:** checkpoint **or** exit — `midway $13CE` rising-edge to `1`, **or**
  `$1F2E` (ExitsCompleted) increments.

**ExitCounter** (refactored from v0.1.1's exitMode-only counter, `Id = "exits"`):
- **Collect:** `exitMode $0DD5` shifts to a value `!= 0` and `!= 128`
  (goal / orb / key / tape — what v0.1.1 already counts).
- **Bank:** `$1F2E` (ExitsCompleted) increments — the exit actually written to
  the save file. Banks only on real completion, **not** on checkpoint.

Why `$1F2E` for banking is safe (it caused the v0.1.0 save-load bug when used as
the *counter*): on attach the baseline is set to the current value, so a
save-load `0→24` populate just sets the baseline (no increment seen); and banking
is `saved = total`, idempotent whenever nothing is pending (`total == saved`).
The **collect** is driven by the event byte (exitMode), never by `$1F2E`, so the
phantom-exit bug cannot return.

Net for exits: visible count still equals *completed* exits, after-goal deaths no
longer inflate it, and save-load still can't phantom-count.

## Display / alert color (both banked counters)

The value uses the **layout's own theme colors** (auto-matching, user-overridable
via LiveSplit layout settings — no custom UI):
- `total == saved` → `state.LayoutSettings.TextColor` (normal).
- `total != saved` → `state.LayoutSettings.BehindLosingTimeColor` (negative color)
  — powerups: a whole-level "carrying, must die before saving" warning; exits:
  the negative color for the brief goal→bank window. This is a static color
  state (the value is simply drawn in the alert color while `total != saved`),
  not an animation. Reverts to normal on bank or on an after-goal death.

## `BankedCounter` base + rendering hook

- New `abstract class BankedCounter : ISmwCounter` holding `total`/`saved`,
  `Value => total`, `ValueIsAlert => total != saved`, the shared **die-revert**
  (`$0071`→9 → `total = saved`), `Reset`, and `SaveState`/`LoadState` (persist
  both values). It reads `$0071` once per poll (for die-revert) and exposes it to
  the collect hook so subclasses don't re-read. Subclasses implement:
  - `DetectCollect(...)` → true when a collect fired,
  - `DetectBank(memory)` → true when a bank fired,
  - detector-baseline clearing on detach.
- `ISmwCounter` gains `bool ValueIsAlert => false;` (default false; existing
  DeathCounter/MoonCounter/JumpCounter keep normal color).
- `SmwCountersComponent.DrawGeneral`: the value color becomes
  `c.ValueIsAlert ? state.LayoutSettings.BehindLosingTimeColor : textColor`
  for that counter's value cell.

## Serialization compatibility (ExitCounter)

v0.1.1 saved only `<Exits>` (the value). v0.2.0 saves both `total` and `saved`.
`LoadState` must stay back-compatible: `total = ParseInt("Exits")`, and
`saved = ParseInt("ExitsSaved", total)` — i.e. if the saved element is absent
(old layout), assume the loaded value was already banked. Element names:
`Exits` (= total) and `ExitsSaved`. Powerups use `Powerups` / `PowerupsSaved`.

## Jump counter fix

`JumpCounter.Poll` gets the same in-level gate as the v0.1.1 moon fix: when
`levelStart $1935 != 1`, clear its baselines and return without counting. Fixes
overworld/load false counts.

**Known limitation (jumps + powerup collect):** the SMW title-screen attract
*demo* may report `levelStart == 1` while a scripted Mario acts, so the gate may
not suppress demo counts. The primary guard remains "only count while the
LiveSplit timer is running," so this only bites if the timer runs on the title
screen (the user's observed 14-jump case). Acceptable for v0.2.0; revisit if it
recurs.

## Testing (TDD, fake ISnesMemory)

`BankedCounter`/`ExitCounter`/`PowerupCounter`:
- **Exit:** collect on exitMode valid-shift (total++, alert on); bank on `$1F2E`
  increment (saved=total, alert off); after-goal death (collect then `$0071`→9
  before `$1F2E`) reverts total, no net count; save-load `$1F2E` `0→24` with no
  pending collect produces no count and no phantom bank; exitMode 128 ignored.
- **Powerup:** collect in-level bumps total; collect off-level does not; die
  reverts total to saved; bank via midway (`$13CE`→1); bank via exit (`$1F2E`
  increment); full low% loop (grab→alert; die→revert, alert off; grab→reach
  midway→banked, saved advanced, alert off).
- `ValueIsAlert` true iff `total != saved` for both.
- ExitCounter `LoadState` back-compat: old `<Exits>` only → total==saved==value.
`JumpCounterTests`: jump edge off-level does not count; in-level does.

## Non-goals

- Custom color settings UI (theme colors already cover it).
- Powerup *state* tracking (only collection + save/death events).
- Changing the 50px row-height default (user is fine with it).
- SNESOffset shared-architecture work (separate effort).

## Success criteria

1. Powerups: collecting shows `total` in the negative color; dying before a save
   reverts it; a checkpoint or exit banks it (alert clears, `saved` advances).
2. Exits: after-goal death does not inflate the exit count; save-load produces no
   phantom exit; normal completions still count once.
3. Both banked counters draw the value in the theme negative color while
   `total != saved` (static color state, not an animation).
4. Jumps/powerups don't count from overworld/load transitions.
5. Powerups off by default, mushroom icon renders, both tallies survive a layout
   save/restore, and old v0.1.1 layouts load without losing the exit count.
6. Version 0.2.0; all unit tests pass.
